## [2026-06-22] S10: Agent 审查与公开提交修正

**目的**：请 AI 以助教审查视角检查工程交付风险，修复公开仓库提交前的问题。

**输入**：当前 `src/` 实现、`docs/` 六类证据文件、测试记录、git 历史。

**建议**：
- AI 指出 `answer.py` 中仍保留个人 API key 默认值，公开仓库存在安全风险。
- AI 指出资料外问题虽然回答了"资料中没有找到依据"，但早期输出仍可能附带 `[faq-XX]` 来源，形成证据链矛盾。
- AI 建议把中文单字 token 改为 2-gram，并过滤"什么/如何"等泛化疑问词。
- AI 建议同步更新 spec、design、test-record、README.交付版.md，避免文档仍描述旧行为。

**人工判断**：
- 采纳移除真实 API key。理由：本地为了验证真实接口可以使用个人 key，但公开提交必须只保留环境变量入口；否则即使功能正确，也不符合基本工程安全要求。
- 采纳"拒答时不显示来源"。理由：拒答的含义是资料中没有可靠依据，如果同时给出 `[faq-XX]`，助教无法判断系统到底相信哪条证据。
- 部分采纳检索调整。只用提高阈值会伤害正常问题召回，因此采用"2-gram + 泛词过滤 + 小范围课程交付同义扩展"的组合方案，保留小项目的简单性。
- 采纳文档同步。理由：工程完整性不是代码单点正确，而是 spec、design、测试记录、README 和实际行为互相印证。

**验证**：
- `test_basic.py` 9/9 通过。
- "奖学金政策是什么？""今天中午吃什么？"返回拒答且 sources 为空。
- 检查仓库中不再出现真实 `sk-` API key。
