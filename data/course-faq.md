## [faq-01] 什么是"可复核交付"？

**可复核交付**是 Day1 训练营的核心评价标准。它不是一个技术概念，而是一种提交方式：

- **定义**：助教不需要看你的代码写得好不好，也不需要问你任何问题，只读你提交的文件就能独立判断你是否达到了训练营要求。
- **核心原则**：提交包的证据足以被第三方独立判断——就像论文审稿，审稿人不认识你、不问你问题，只看你写的东西就能打分。
- **反例**：只提交代码、没有 README、没有 AI 协作记录——助教不知道你的代码能不能跑、AI 帮你做了什么、你自己做了什么判断。
- **正例**：提交包包含 6 类标准文件——助教按 README 运行你的代码，按 test-record 验证结果，按 ai-log 判断你的 AI 协作质量。

在训练营中，PASS/WARNING/BLOCKED 三级评价全部基于提交包的证据完整度，与代码写得"好不好"无关。

## [faq-02] 6 类证据文件分别是什么？

Day1 最终提交包必须包含 6 类标准文件：

| 文件 | 用途 | 谁来判断 |
|------|------|---------|
| `spec.md` | 定义要做什么、不做什么、怎么验收 | 助教看 Spec 就知道你的项目范围 |
| `design.md` | 技术方案和数据流设计 | 助教看设计就知道你的实现思路 |
| `ai-log.md` | 整个开发过程中与 AI 的协作决策记录 | 助教看 AI log 就知道你做了哪些判断 |
| `test-record.md` | 测试输入、预期输出、实际输出 | 助教看测试记录就能验证功能正确性 |
| `README.md` | 安装步骤、运行命令、测试命令 | 助教照 README 就能复现你的项目 |
| `reflection.md` | 复盘：认知变化、遇到的困难、改进方向 | 助教看复盘就知道你学到了什么 |

**不是**：把代码写得很漂亮、用最新框架、实现复杂功能。
**而是**：让一个不认识你的人，只看这些文件就能完整判断你的工作。

## [faq-03] ai-log 的五字段格式是什么？

ai-log 是记录每一次与 AI 协作决策的结构化日志，每一条包含五个字段：

| 字段 | 含义 | 示例 |
|------|------|------|
| **目的** | 这次问 AI 想达成什么 | "请 AI 审查我的 Spec 是否有遗漏" |
| **输入** | 给 AI 提供了什么材料/约束 | "提供了 course-faq.md 和当前 spec.md 草稿" |
| **建议** | AI 给出了什么方案 | "AI 建议增加用户登录功能和数据库存储" |
| **人工判断** | 你采纳/修改/拒绝了 AI 的什么建议，为什么 | **"拒绝数据库——课程要求 CLI 工具，不能引入外部依赖"** |
| **验证** | 如何证明最终结果可靠 | "用 course-faq.md 的 10 个问题手动测试，回答全部正确" |

**最关键字段是"人工判断"**——如果整条 ai-log 没有你的判断（只是"AI 说的都对"），这条记录在评审中是无效的。

## [faq-04] Spec 的三要素是什么？

Spec（规格说明）是 AI Native 开发的第一步：在写代码之前，先用文档定义清楚做什么。三要素：

1. **目标（Goals）**：项目要完成什么，≤5 条，每条可验证。
   - 例："用户输入问题后，CLI 输出基于 course-faq.md 的回答和来源编号"
2. **非目标（Non-Goals）**：明确不做什么，≥3 条，每条对应一个 AI 可能建议的过度设计。
   - 例："不做 Web UI"、"不做用户登录"、"不做数据库存储"、"不支持 PDF 上传"
3. **验收标准（Acceptance Criteria）**：≥3 条可测试的通过条件。
   - 例："输入'Day1 要交什么？'，输出包含 6 类证据文件的描述和来源[faq-02]"

**为什么需要 Spec**：没有 Spec，AI 会做过度设计——你问"帮我做课程助手"，AI 可能给你一个带登录、数据库、Web UI 的完整系统。有了 Spec（特别是非目标），AI 的行为被约束在明确边界内。

## [faq-05] RAG 是什么？为什么 Day1 要自己实现 RAG？

**RAG（Retrieval-Augmented Generation）** 是"检索增强生成"：先检索相关资料，再把资料和问题一起发给 LLM，让 LLM 基于资料回答——而不是让 LLM 凭记忆回答。

**RAG 数据流六环节**：

```
资料源 → 切片 → 检索 → Prompt 拼接 → 来源引用 → 资料外拒答
```

- **资料源**：course-faq.md 是你的知识库
- **切片**：把 FAQ 文件按 `##` 标题切成独立 chunk，每个 chunk 有唯一 ID（如 `faq-03`）
- **检索**：用户问题 → 关键词分词 → 匹配每个 chunk → 计算命中分数 → 排序 → 返回 Top3
- **Prompt 拼接**：系统指令（"只基于资料回答"）+ 检索到的 Top3 chunk + 用户问题
- **来源引用**：回答必须注明引用哪个 chunk（如"来源：[faq-03]"）
- **资料外拒答**：问题与资料无关时，拒绝回答并提示"资料中没有找到依据"

**为什么自己做而不直接用 LLM 的文件功能**：自己做一遍 RAG 才能理解检索机制（关键词匹配、阈值调优）、控制来源引用（确保回答可追溯）、做到真正的资料外拒答（而非 LLM 自由发挥）。Day1 的目标不是做一个比 LLM 更聪明的工具，而是理解 RAG 背后的运作机制。

## [faq-06] Day1 的工具链是什么？

Day1 要求跑通的工具链：**建目录 → 写文档 → Git → 提交**。

具体步骤：
1. 用 `mkdir -p rag-assistant/{src,data,docs,tests}` 创建标准项目骨架
2. 编写 README.md（安装/运行/测试命令）和 CLAUDE.md（AI 协作上下文）
3. 用 `git init && git add -A && git commit -m "有意义的提交信息"` 建立 Git 版本控制
4. 每个 commit 必须有意义的 message，不能是 "update" 或 "fix"
5. 提交前运行检查清单：目录齐全 / README 可复现 / AI log 有判断 / 产物完整

**备选方案**：如果 Git 不可用，使用 `tar -czf rag-assistant-backup.tar.gz rag-assistant/` 打包提交。

工具链的核心不是"会用这些命令"，而是**建立可追溯的工程纪律**——每一行代码、每一个 commit、每一条 AI 决策都有记录，可以被独立评审。

## [faq-07] Bug 修复的五步法是什么？

Bug 修复不是"看到报错直接丢给 AI 修"，而是有严格的方法论——**复现 → 定位 → 假设 → 最小修复 → 验证**：

| 步骤 | 做什么 | 产出 |
|------|--------|------|
| **复现** | 写 reproduction.md：输入、期望输出、实际输出、环境信息 | 一个别人能照着重现的 Bug 报告 |
| **定位** | 缩小范围到具体文件、函数、行号 | root-cause.md 中的可疑位置 |
| **假设** | 提出至少 2 个可验证的根因假设 | 每个假设附带验证方法 |
| **最小修复** | 只改最少行数（≤5 行） | git diff 只有必要的改动 |
| **验证** | 跑回归测试（正常/边界/异常三类输入） | fix-record.md 中的测试结果 |

**为什么不能直接让 AI 批量修**：AI 可能改对 Bug，但顺便改了无关代码（引入新 Bug），或者给出过度修复方案。五步法的核心是**人负责诊断和决策，AI 辅助验证和执行**——和写 Spec 一样，先用文档定义问题范围，再动手。

## [faq-08] 提交前检查清单是什么？

在每次提交（或每节课结束）前，逐项检查以下内容：

1. **目录齐全**：`rag-assistant/src/`、`data/`、`docs/`、`tests/` 存在且非空
2. **README 可复现**：包含 `python3 src/main.py`、测试命令，助教照做能跑通
3. **AI log 有判断**：每条 ai-log 的"人工判断"字段不是空的、不是"AI 说的都对"
4. **产物完整**：对照当天应产出文件列表，逐项核对

**检查方式**：
- 手动核对：打开每个文件，确认内容不是模板占位符
- 或运行 `check-submission.sh` 自动检查

提交前检查不是"看看就好"，而是**提交包的最后一个质量关卡**——如果检查清单上有任何一项不合格，提交就不应该发生。

## [faq-09] 五维能力画像是什么？

五维能力画像是 Day1 训练营的评估框架，在 S1 导入、贯穿全天、S8 自评。每维 1-5 分，有明确的行为锚点：

| 维度 | 1 分（初始） | 5 分（目标） |
|------|-------------|-------------|
| **上下文工程** | 直接把需求丢给 AI | Spec 有目标/非目标/验收，AI 能直接执行 |
| **工具链熟练度** | 需助教一步步指导 | 独立跑通闭环，commit 有意义 |
| **诊断修复** | 报错直接丢给 AI 修 | 先写 reproduction，最小修复，补回归测试 |
| **交付完整度** | 只有代码 | 6 类文件齐全，助教只读文件就能判断 |
| **RAG 理解应用** | 不理解 RAG 原理 | 能设计数据流，实现可运行 CLI RAG MVP |

**如何使用**：
- S1：导入这五个维度，让学生知道今天学什么、被评什么
- S8：对照五个维度自评，每维引用具体文件作为证据（如"上下文工程 4 分，证据：spec.md 写了 4 个目标和 5 个非目标"）
- 自评不是"我觉得我学得不错"，而是逐维用证据说话

## [faq-10] 什么是"非目标"？为什么 Spec 需要它？

**非目标（Non-Goals）** 是 Spec 的第二要素，明确列出项目**不做什么**。

**为什么需要非目标**：
- AI 天然倾向过度设计——当你问"帮我做课程助手"，AI 可能建议：用户登录、MySQL 数据库、Web UI、PDF 导出、权限系统……
- 没有非目标，你无法拒绝这些建议——因为它们"听起来都合理"
- 有了非目标，你可以说"这个不在 Spec 的非目标列表里"——这是基于设计决策的拒绝，而不是基于个人喜好

**典型非目标示例**（课程 RAG 助手项目）：
- 不做 Web UI（CLI 工具）
- 不做用户登录（单用户本地运行）
- 不做数据库存储（文件系统就是数据源）
- 不支持 PDF 上传（只支持 Markdown）
- 不支持除 course-faq.md 之外的资料源

非目标的数量 ≥3 条，每条对应一个 AI 可能建议的常见过度设计方向。写好非目标的关键不是列举，而是**预判 AI 会建议什么并提前封堵**。
